# 🕶️ Dark Panel

پنل مدیریت چندسطحی برای ساخت کانفیگ‌های VLESS / Trojan (سازگار با V2Ray، Xray، Hiddify، v2rayNG، Streisand و...) روی Railway.
ساخته‌شده توسط **adel**.

## ساختار دسترسی

- **سوپر ادمین**: با ایمیل و رمز عبور وارد می‌شود. می‌تواند ادمین (نماینده) جدید بسازد، تعداد روز اعتبار و سقف تعداد کانفیگ هرکدام را تعیین کند، تمدید/غیرفعال‌سازی/حذف کند.
- **ادمین (نماینده)**: با ایمیلی که سوپر ادمین برایش ساخته وارد می‌شود و فقط در بازه‌ی اعتبار و سقف کانفیگ تعیین‌شده می‌تواند کانفیگ بسازد.
- ثبت‌نام عمومی وجود ندارد — فقط ورود با حسابی که سوپر ادمین ساخته.

اطلاعات ورود پیش‌فرض سوپر ادمین (قابل تغییر با متغیرهای محیطی `SUPERADMIN_EMAIL` / `SUPERADMIN_PASSWORD`):

```
ایمیل:  mohammadadel.zare@gmail.com
رمز:    Adel5129
```

## انواع کانفیگ

هر ادمین می‌تواند از این نوع کانفیگ‌ها بسازد (همه با کلاینت‌های V2Ray/Xray/Hiddify کار می‌کنند):

- VLESS · WebSocket
- VLESS · XHTTP (packet-up)
- VLESS · XHTTP (stream-up)
- Trojan · WebSocket

برای هر کانفیگ قابل تنظیم است: برچسب/نام، حجم مجاز (KB/MB/GB)، تعداد روز اعتبار، محدودیت سرعت (Mbit/s)، محدودیت تعداد آی‌پی هم‌زمان، پورت، Fingerprint و ALPN.

### ⚠️ محدودیت‌های فنی مهم (حتماً بخوانید)

- **OpenVPN واقعی روی Railway اجرا نمی‌شود.** Railway اجازه‌ی ساخت دیوایس TUN/TAP یا حالت privileged شبکه را نمی‌دهد که OpenVPN بدون آن‌ها بالا نمی‌آید. به‌جای آن، پروتکل‌های VLESS و Trojan (که پایه‌ی V2Ray/Xray/Hiddify هستند) پیاده‌سازی شده‌اند و روی Railway کامل کار می‌کنند.
- **VMess در این نسخه پیاده‌سازی نشده.** رمزنگاری AEAD/legacy مربوط به VMess پیچیده و پرخطا است؛ برای اطمینان از درست‌کار کردن، فعلاً فقط VLESS (بدون رمزنگاری در این لایه، متکی به TLS خودِ Railway) و Trojan (که ساده‌تر و امن‌تر قابل پیاده‌سازی بود) اضافه شدند. در صورت نیاز، VMess می‌تواند در به‌روزرسانی بعدی اضافه شود.
- **ربات مدیریت تلگرام از این نسخه حذف شد** چون کاملاً برای مدل تک‌ادمین قبلی نوشته شده بود و با مدل چندسطحی جدید سازگار نبود. در صورت نیاز می‌توان دوباره برایش نسخه‌ی چندسطحی نوشت.

## Deploy روی Railway

1. این ریپازیتوری را Fork کنید.
2. در [Railway.app](https://railway.app) → New Project → Deploy from GitHub repo → ریپازیتوری Fork‌شده را انتخاب کنید.
3. یک **Volume** بسازید و آن را روی مسیر `/data` وصل کنید (تنظیمات سرویس → Volumes) تا اطلاعات (ادمین‌ها، کانفیگ‌ها، رمزها) با هر ری‌استارت پاک نشوند.
4. یک دامنه‌ی عمومی (Public Domain) برای سرویس فعال کنید.
5. به آدرس `https://your-app.up.railway.app/login` بروید و با ایمیل/رمز سوپر ادمین وارد شوید.

| `DATA_DIR` | مسیر ذخیره‌سازی وضعیت (نیازمند Volume دائمی) | `/data` |
| `RAILWAY_PUBLIC_DOMAIN` | دامنه‌ی عمومی سرویس؛ خودکار توسط Railway تنظیم می‌شود | `localhost` |

## جریان کار

1. با حساب سوپر ادمین وارد شوید.
2. از بخش «مدیریت ادمین‌ها»، یک ادمین/نماینده جدید با ایمیل، رمز عبور، تعداد روز اعتبار و سقف تعداد کانفیگ بسازید.
3. آن ایمیل/رمز را به نماینده بدهید تا با آن وارد پنل خودش شود.
4. نماینده از پنل خودش کانفیگ‌های VLESS/Trojan با حجم و روز دلخواه می‌سازد و لینک/QR هرکدام را به مشتری می‌دهد.
5. وقتی اعتبار روزانه‌ی نماینده تمام شود یا سوپر ادمین او را غیرفعال/حذف کند، تمام کانفیگ‌های او هم به‌طور خودکار از کار می‌افتند.
